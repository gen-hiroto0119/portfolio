## 背景と課題

クライアントの業務SaaSは、売上・在庫・サポート指標がそれぞれ別画面に存在していました。経営層は週次レビューのたびに複数タブを行き来し、スクリーンショットをSlackに貼って議論する運用が常態化していました。

現場ヒアリングでは「数字はあるが、次に何をすべきかが見えない」という声が繰り返し上がりました。ダッシュボードは見た目の問題ではなく、情報設計とアクション導線の問題だと判断しました。

## 意思決定

まず指標を「監視」「診断」「アクション」の3層に分類し、画面上部には監視指標のみを配置しました。異常値が出たときだけ診断パネルが展開されるインタラクションにすることで、常時表示の情報量を抑えています。

色の意味付けはアクセントカラーを警告用途に限定し、通常状態はモノトーンで統一しました。これにより、本当に注意が必要な箇所だけが目に入る設計になっています。

## 技術選定

フロントエンドは既存基盤の React を活かしつつ、新ダッシュボード部分のみ Next.js App Router で段階的に置き換えました。スタイリングはデザイントークンを StyleX で一元管理し、コンポーネント間の視覚的一貫性を担保しています。

データ取得は TanStack Query を採用し、ウィジェット単位でキャッシュ戦略を変えることで初期表示を高速化しました。重い集計APIはサーバー側でプリフェッチし、クライアントには整形済みデータのみ渡す構成です。

```tsx
export async function DashboardPage() {
  const overview = await getOverviewMetrics();
  return (
    <DashboardShell initialData={overview}>
      <KpiStrip metrics={overview.kpis} />
      <DiagnosticPanel triggers={overview.alerts} />
    </DashboardShell>
  );
}
```

## 成果

リリース後6週間で、ダッシュボードの日次利用者数は38%増加しました。週次レビューでは「数値確認」に費やす時間が大幅に減り、議論の中心が対策案の比較に移っています。

定量指標に加え、現場マネージャーからは「朝イチで異常箇所だけ確認できる」という運用面の評価も得られ、継続的な改善バックログの優先度付けにもつながりました。
